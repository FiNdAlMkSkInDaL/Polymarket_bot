===============================================================================
POLYMARKET BOT - SYSTEM EXPLANATION
Last updated: March 20, 2026
===============================================================================

This repository is no longer a bar-driven directional trading bot.

The live system is now a high-frequency pure market maker and latency
arbitrage engine for Polymarket's CLOB. Any reader, operator, or future AI
agent should treat that as the primary architectural truth.


1. CURRENT REALITY
===============================================================================

The active production architecture has three live strategy families:

1. Pure Market Maker
2. SI-9 Combinatorial Arbitrage
3. Latency Arbitrage
   - SI-7 crypto fast-strike
   - SI-8 news/oracle fast-strike

The old directional stack is retired for live trading.

Retired from live architecture:
- PanicDetector-driven entries
- Mean-reversion drift entries
- bar-entry logic as the primary execution path
- deployment decisions based on legacy z-score or drift-ceiling style knobs

Some of that code still exists for research, compatibility, and legacy replay,
but it is not the live engine. If a document, script, or config discussion
frames this repo as a panic-fading or bar-signal bot, that description is
wrong.


2. EXECUTIVE SUMMARY
===============================================================================

The bot is a Polymarket microstructure system that does two things well:

- Rest passive liquidity where the spread is worth capturing.
- Hit stale liquidity when an external information edge appears before the
  CLOB reprices.

In practice that means:

- full L2 order book reconstruction on the top active markets
- passive NO-token quoting for spread capture
- maker-first execution for mutually exclusive event arbitrage
- taker fast-strike execution when crypto spot or breaking news moves faster
  than the Polymarket book
- strict latency, liquidity, and toxicity gates around every entry path


3. HIGH-LEVEL RUNTIME ARCHITECTURE
===============================================================================

3.1 Data plane

The live bot consumes and maintains:

- Polymarket trade and book streams
- dedicated L2 snapshot and delta data
- optional external price and oracle feeds for latency arbitrage

The system load-sheds aggressively:

- the live paper deployment hard-caps the tracked market universe at 25 markets
- those tracked markets are selected by volume and reconciled on refresh so the
  bot does not silently grow back to a 100+ market event-loop workload
- full L2 reconstruction runs only on the top tracked markets inside that cap

Current operator workflow for that universe cap:

- scripts/screen_wick_markets.py is the offline pre-trade universe screener for
  the stink-bid / wick-catching research path
- it ranks active Gamma markets by a simple wick-potential proxy:
  24h volume divided by resting liquidity
- the terminal table is for human review and may show the numeric Gamma market
  id for readability
- if --export-json is used, the script must export Polymarket CLOB-compatible
  conditionId hex strings, not Gamma numeric ids
- that JSON export is the correct handoff format for config injection because
  the live bot, subscriptions, and L2 paths key markets by conditionId-style
  identifiers rather than Gamma's numeric primary key
- scripts/screen_si9_clusters.py is the offline discovery screener for the
  SI-9 mutually exclusive negRisk cluster path
- it queries Gamma /events, filters to active open negRisk-compatible event
  groups, and ranks clusters by aggregate 24h volume divided by aggregate
  resting liquidity
- its JSON export preserves per-leg conditionId hex strings for each cluster so
  SI-9 downstream consumers stay keyed to CLOB-compatible identifiers rather
  than Gamma numeric market ids

L2 processing is first-class infrastructure, not an optional add-on.
Worker processes can reconstruct books off the main asyncio loop and publish
state back through shared memory so strategy code reads current BBO and depth
without blocking the bot.

3.2 Strategy plane

The strategy layer is centered on three active paths:

- PureMarketMaker in src/strategies/pure_market_maker.py
- SI-9 combo arb detection and execution routing
- SI-7 and SI-8 latency-arb routes that flow through the fast-strike taker
  path in the position manager

3.3 Execution and risk plane

Execution is split by intent:

- passive quoting and maker routing for spread capture
- immediate taker routing for stale liquidity sniping
- stateful position management, stop logic, and adverse-selection defenses

Risk enforcement is continuous:

- latency guard
- L2 reliability checks
- adverse-selection fast-kill logic
- inventory caps for market making
- wallet and exposure caps

Current sizing workflow:

- pure_mm_wide_size_usd is a live/replay configuration knob for the wide quote
  tier
- that size is not yet optimized directly inside the current WFO search grid
- the standalone helper scripts/kelly_sizer.py is now the intended offline
  calculator for translating WFO win-rate and payoff stats into a bankroll-
  aware pure_mm_wide_size_usd recommendation
- the script prints both full-Kelly and half-Kelly outputs, with half-Kelly as
  the intended default risk-management posture

Current live paper thresholds were deliberately relaxed after VPS runtime
verification showed normal Polymarket jitter could otherwise trip the breaker:

- LATENCY_BLOCK_MS = 5000
- HEARTBEAT_STALE_MS = 15000

Those values are now part of the intended PAPER deployment posture, not an
accidental local override.

3.4 Monitoring and recovery

The system records fills and checkpoints, logs structured events, sends
Telegram alerts, and relies on circuit breakers around hot loops. If a core
loop repeatedly fails, the bot suspends and resets instead of continuing in a
half-broken state.

Current deployment reality:

- the live paper bot on the VPS is now managed by systemd
- the canonical service is polymarket-bot.service
- the service runs src.cli in PAPER mode, not an ad-hoc nohup shell
- Restart=always is part of the intended operating model so the process comes
  back after crashes or host reboots
- service stdout and stderr stay in native journald
- bot.log is not the canonical live operations log and should not be treated as
  the source of truth for current system state

Operational implication:

- if you are restarting, checking health, or reviewing uptime on the VPS,
  systemctl and journalctl are the source of truth
- if journal output and bot.log disagree, trust journalctl
- a stray nohup-style background process should be treated as operational
  drift, not the desired deployment state

Current latency telemetry workflow:

- scripts/profile_latency_logs.py is the standalone offline profiler for raw
  text exports captured from journald
- it is intended to parse processing-latency and heartbeat-delta style events
  from exported journal output rather than from legacy bot.log files
- its default histogram view includes a red 5000 ms block-threshold marker so
  operators can see the current live paper safety margin against the latency
  guard cutoff


4. ACTIVE STRATEGY 1 - PURE MARKET MAKER
===============================================================================

This is the primary live engine.

What it does:

- selects the highest-volume tracked markets, with the live tracked universe
  currently capped at 25
- quotes the NO token passively
- posts a best-bid-style passive buy quote
- only posts the ask when inventory already exists
- cycles inventory from filled bid to passive ask to capture spread

Important implementation details:

- quoting is inventory-aware, so the engine does not rely on naked asks
- quote size is capped by a per-market-maker inventory budget
- selected markets come from the capped active tracked universe, not from bar
  signals
- startup and refresh both reconcile the tracked set so universe bloat does not
  reappear after discovery refreshes
- the engine runs in a tight loop suitable for HFT quote maintenance

Why it can make money:

- the bot earns spread when a passive bid fills and that inventory is later
  distributed through a passive ask
- the strategy is designed for high-liquidity binary books where queue
  position, spread width, and quote survival matter more than slow-moving
  directional forecasts

Why it can avoid getting run over:

The bot does not leave passive quotes resting blindly. It cancels when the
microstructure turns toxic. The current live design explicitly uses:

- OFI, or order flow imbalance
- depth evaporation over a short time window
- L2 reliability gates
- latency health and fast-kill conditions
- optional iceberg and venue-microstructure cues from the broader system

The key idea is simple: capture passive spread when flow is benign, and cancel
before informed flow runs through the quote.


5. ACTIVE STRATEGY 2 - SI-9 COMBINATORIAL ARBITRAGE
===============================================================================

SI-9 is a maker-first state machine for mutually exclusive market clusters.

When a Polymarket event exposes a set of mutually exclusive outcomes and the
cluster is marked negRisk-compatible, the bot checks whether the combined YES
pricing is inconsistent with a guaranteed payout structure.

Core idea:

- if mutually exclusive outcomes are mispriced, the bot can buy a balanced set
  of legs whose combined settlement value is locked in

Current live design:

- clusters are built from mutually exclusive event groups
- the hardest leg to fill is chosen as the maker leg
- that bottleneck leg is worked passively first
- the remaining legs are routed after the maker-first state machine confirms
  the opportunity can be completed defensibly

This is not a generic stat-arb module. It is a routing-aware execution system
for Dutch-book style mispricings in mutually exclusive Polymarket structures.


6. ACTIVE STRATEGY 3 - LATENCY ARBITRAGE
===============================================================================

Latency arbitrage is the taker side of the architecture. When an external
source of truth moves faster than the Polymarket book, the bot attempts to hit
stale liquidity immediately.

There are two live latency-arb flavors.

6.1 SI-7 - Crypto Fast-Strike

SI-7 targets crypto-linked prediction markets.

What drives it:

- free external crypto spot feeds for BTC and ETH
- the Resolution Probability Engine in src/signals/resolution_probability.py
- Black-Scholes-style threshold pricing for crypto strike markets
- a fast retrigger loop that re-evaluates crypto markets whenever spot moves
  enough to create a fresh divergence

How it executes:

- when the model detects a sufficiently stale Polymarket price and latency is
  healthy, the position manager can use the fast-strike taker path
- those positions intentionally skip passive chasing because the edge is the
  speed advantage itself

Operational meaning:

- SI-7 is not slow research alpha
- it is an HFT stale-order sniping path driven by free crypto market data

Offline calibration support:

- scripts/calibrate_si7_pricing.py is the standalone SI-7 research helper for
  plotting Black-Scholes in-the-money probability curves, specifically N(d2),
  around a fixed strike
- it is meant for offline calibration of how Polymarket-style probability
  should move as spot approaches strike under different implied-volatility
  regimes
- it does not replace src/signals/resolution_probability.py and is not part of
  the live execution path

6.2 SI-8 - News Arbitrage

SI-8 is the generalized oracle/news fast-strike path.

Current live deployment focus:

- Tree News WebSocket firehose for breaking news and resolution-relevant event
  updates

How it works:

- off-chain adapters emit OracleSnapshot objects
- OracleSignalEngine converts verified external state changes into a market
  probability view
- if Polymarket is stale and spread and confidence gates pass, the bot routes
  the trade into the fast-strike execution path

Zero-overhead architecture rule:

- paid non-crypto and paid sports feeds are hard-disabled in the intended live
  deployment model
- the repo still contains adapter scaffolding for sports and Odds API style
  integrations, but those are not part of the target production architecture
- the production direction is zero-overhead latency arb built on free crypto
  spot data and free Tree News WebSocket data

If you are configuring live SI-8 today, the intended production assumption is:

- Tree News is active
- paid The Odds API style paths are not part of the zero-overhead live stack


7. DATA AND MARKET MICROSTRUCTURE REQUIREMENTS
===============================================================================

The architecture pivot changed the data requirements completely.

Trade receipts alone are no longer enough to understand or simulate the live
bot.

The current live system depends on L2 information for:

- queue-aware passive quoting
- best bid and best ask evolution
- OFI calculation
- depth-evaporation detection
- maker fill realism
- stale-book and toxicity gating

For the current bot, L2 is not optional historical enrichment. It is core
strategy data.

Current recording format:

- live forward-test data is written under data/raw_ticks/YYYY-MM-DD/
- each asset writes a JSONL file of raw recorded events
- the recorder now persists both:
  - ongoing price_change L2 deltas from the WebSocket path
  - authoritative full-depth book snapshots fetched through the REST L2 sync
    path

This distinction matters. Earlier forward data could contain only deltas in a
sampled slice, but the hardened live recorder is expected to capture both
event families over the course of normal operation.


8. BACKTEST AND WFO PIVOT
===============================================================================

Walk-Forward Optimization has been realigned around the new pure market maker
architecture.

Important facts:

- WFO now supports a pure market maker replay adapter
- that replay path depends on L2 snapshots and L2 deltas
- pure market maker WFO is invalid on trade-only datasets

Why trade-only data is insufficient:

- passive maker fills cannot be simulated defensibly from trade receipts alone
- OFI cannot be reconstructed from trade-only data
- depth evaporation cannot be measured from trade-only data
- quote cancellation logic cannot be replayed faithfully without book changes

Current enforced behavior:

- when WFO is run with strategy_adapter set to pure_market_maker and the
  replay sees zero L2 book updates, the optimizer logs a missing-L2 warning
  and returns no usable result

This is deliberate. A fake pure-MM backtest on trade-only data is worse than
no backtest because it produces parameter outputs that look scientific but do
not reflect the live execution problem.

If you want valid pure market maker backtests, your dataset must contain:

- L2 snapshots
- L2 deltas
- correctly ordered book events over time

Current operational verification path:

- the script scripts/audit_forward_data.py is the forward-data health check
- it is designed to inspect recent live raw-tick files and verify that fresh
  data contains both snapshots and deltas
- the strict gate is --require-wfo-ready
- passing that audit is the intended proof that the live paper recorder is
  capturing replayable pure-MM data

Current optimizer grid state:

- the WFO search space is now pre-wired for the wide liquidity tier
- pure_mm_wide_tier_enabled is forced on in the pure-MM WFO path so the
  optimizer actually exercises the Tier-2 quote logic
- pure_mm_wide_spread_pct is bounded to 0.05 through 0.25 for the Monday
  sweep
- pure_mm_toxic_ofi_ratio is narrowed to 0.60 through 0.95
- the focused pure-MM WFO entrypoint at repo root now includes the wide-tier
  activation flag and wide spread parameter in its target parameter set

Important limitation:

- pure_mm_wide_size_usd exists in live and replay configuration, but it is not
  yet part of the current optimizer sweep because no explicit search bounds
  have been locked for it in this repo state
- position sizing for that parameter is currently a post-WFO operator step,
  not an in-search optimization dimension

Current offline analysis tooling for that sweep:

- scripts/visualize_wfo_surface.py is the standalone parameter-surface analysis
  script for the pure-MM wide-tier grid
- it accepts JSON or CSV exports that contain pure_mm_wide_spread_pct,
  pure_mm_toxic_ofi_ratio, and a profit-style metric such as PnL, EV, or
  objective score
- it can render a static matplotlib heatmap, a static 3D surface, or both
- it also supports standalone Plotly HTML export for browser-based rotation,
  zoom, and hover inspection of the same parameter surface
- the hover contract in that HTML export is intended to show the wide spread,
  toxic OFI ratio, and the selected profit metric for each grid point
- the plateau-scoring view is intentional: the goal is to find broad,
  repeatable profitable regions rather than a single fragile peak

Important nuance:

- for a live continuously appended dataset, the correct audit behavior is to
  sample the recent tail across many recently modified files
- sampling the oldest records or exhausting the sample budget on one hot file
  can produce false negatives even when the system is healthy

Trade-only archives may still be useful for limited legacy research, but they
are not sufficient for the current live pure market making architecture.


8.5 OFFLINE UNIVERSE SCREENING
===============================================================================

The repo now includes dedicated offline market-selection helpers for weekly
universe curation.

Purpose:

- identify markets with high turnover but relatively thin displayed liquidity
- prioritize books that are more likely to suffer sweeps, vacuums, and wicky
  temporary dislocations
- identify mutually exclusive negRisk event clusters suitable for SI-9 maker-
  first combo arbitrage research
- produce a machine-readable universe candidate list without modifying the live
  engine itself

Current tools:

- scripts/screen_wick_markets.py
- scripts/screen_si9_clusters.py

Current scoring rules:

- wick potential = volume_24h / resting_liquidity
- SI-9 cluster score = aggregate cluster volume_24h / aggregate cluster
  resting_liquidity

Current export rules:

- the wick screener JSON export is an array of conditionId hex strings like
  0x...
- the SI-9 cluster screener export is a list of cluster objects whose
  condition_ids and per-leg condition_id fields also preserve those same 0x...
  conditionId strings
- this is intentional because src/bot.py and the Polymarket CLOB/L2 plumbing
  track markets by conditionId-compatible identifiers
- exporting Gamma numeric ids here would create downstream subscription and
  config mismatch risk

Operational implication:

- if you are preparing Monday's tracked-market config from the screener, treat
  the JSON export as the authoritative machine interface
- treat the printed table as a human ranking surface, not the machine contract
- if you are preparing an SI-9 research universe, treat the cluster export as
  the authoritative machine interface for per-leg conditionId membership


9. WHAT FUTURE AGENTS SHOULD ASSUME
===============================================================================

If you are an AI agent reading this file later, start with these assumptions:

- the bot is an HFT pure market maker and latency arbitrage engine
- L2 data is a hard dependency for understanding live behavior
- the VPS paper deployment is expected to run under systemd, not nohup
- the live tracked universe is intentionally capped at 25 markets unless that
  operating assumption is explicitly changed
- the offline market screener for wick-catching exports conditionId hex strings,
  not Gamma numeric ids, when generating machine-readable universe files
- the offline SI-9 cluster screener also exports CLOB-compatible conditionId
  hex strings per leg, not Gamma numeric ids
- the pure-MM WFO surface visualizer is an offline standalone script and is
  not part of the live trading engine or replay execution path
- if interactive inspection is requested, the expected artifact is a standalone
  Plotly HTML export rather than a dependency on the live bot UI
- the SI-7 Black-Scholes pricing calibrator is also offline-only research
  tooling and does not change the live Resolution Probability Engine path
- the live PAPER deployment uses relaxed latency thresholds (5s block, 15s
  heartbeat stale) to tolerate real Polymarket network jitter
- journald, not bot.log, is the authoritative live logging surface
- the intended offline latency review path for live journald exports is
  scripts/profile_latency_logs.py
- pure market maker is the main passive strategy
- SI-9 is the maker-first mutually exclusive combo arb module
- SI-7 and SI-8 are fast-strike stale-book sniping paths
- paid non-crypto and paid sports oracle feeds are not part of the intended
  zero-overhead production architecture
- legacy directional signal code is not the live trading core

Do not center your analysis on retired live concepts such as:

- panic-spike entries
- drift-signal entries
- bar-driven entry pipelines
- z-score-based deployment narratives

Those may still appear in research and replay code, but they do not define the
current system.


10. REPOSITORY MAP FOR THE CURRENT ARCHITECTURE
===============================================================================

Primary files for the live architecture:

- src/bot.py
  Main orchestrator. Wires market data, L2, pure MM, SI-7, SI-8, and SI-9,
  and now hard-caps the tracked live universe to the configured operating
  limit.

- src/core/config.py
  Runtime operating limits and circuit-breaker thresholds, including the live
  paper defaults for tracked market count, latency block threshold, and stale
  heartbeat threshold.

- src/strategies/pure_market_maker.py
  Inventory-aware passive quoting engine.

- src/signals/combinatorial_arb.py
  SI-9 detection and maker-leg routing logic.

- src/signals/resolution_probability.py
  Crypto pricing model and RPE path used by SI-7.

- src/signals/oracle_signal.py
  Oracle snapshot to actionable divergence signal conversion for SI-8.

- src/data/oracle_adapter.py
  Registry and adapter abstractions for external event feeds.

- src/data/adapters/tree_news_websocket_adapter.py
  Tree News WebSocket integration for breaking-news latency arb.

- src/data/l2_book.py
  Full depth reconstruction and L2 state management.

- src/data/l2_websocket.py
  Live L2 snapshot and delta ingestion, plus recording of synchronized REST
  snapshots for forward replay data.

- scripts/polymarket-bot.service
  Canonical systemd unit for VPS deployment. Runs the bot in PAPER mode during
  forward data collection and emits logs to journald.

- scripts/install_paper_service.sh
  One-shot VPS helper that installs/enables/restarts the systemd unit and runs
  a post-start audit.

- scripts/audit_forward_data.py
  Forward-data audit script that verifies live raw JSONL files contain both L2
  snapshots and L2 deltas with real depth arrays.

- scripts/screen_wick_markets.py
  Offline market-universe screener for high-wick-potential books. Ranks active
  markets by 24h volume divided by resting liquidity and can export a JSON
  array of conditionId hex strings for config injection.

- scripts/screen_si9_clusters.py
  Offline SI-9 discovery screener for active mutually exclusive negRisk event
  clusters. Groups legs by Gamma event, ranks clusters by aggregate 24h volume
  divided by aggregate resting liquidity, and exports per-leg conditionId hex
  strings for machine-readable cluster membership.

- scripts/kelly_sizer.py
  Standalone bankroll-aware sizing helper. Converts WFO win probability,
  average win, and average loss into full-Kelly and half-Kelly
  pure_mm_wide_size_usd recommendations.

- scripts/calibrate_si7_pricing.py
  Standalone offline SI-7 calibration helper. Plots Black-Scholes in-the-money
  probability curves using N(d2) across a spot range around strike and overlays
  multiple implied-volatility regimes for calibration work.

- scripts/visualize_wfo_surface.py
  Standalone WFO analysis tool for the pure-MM wide-tier parameter grid.
  Accepts JSON or CSV result exports, plots pure_mm_wide_spread_pct against
  pure_mm_toxic_ofi_ratio, and supports both static matplotlib output and
  standalone interactive Plotly HTML export for plateau inspection.

- scripts/profile_latency_logs.py
  Standalone offline telemetry profiler for exported journald logs. Parses
  latency-like processing and heartbeat delta events from raw text exports,
  prints summary statistics, and renders a histogram with the 5-second block
  threshold marked in red.

- src/backtest/strategy.py
  Contains the PureMarketMakerReplayAdapter used for realistic replay.

- src/backtest/wfo_optimizer.py
  WFO pipeline with pure-MM fail-fast behavior when no L2 data exists and the
  current pure-MM search grid, including wide-tier parameter support.

- wfo_optimizer.py
  Convenience pure-MM WFO entrypoint for the PMXT 7-day dataset with the
  current focused target-parameter list.


11. BOTTOM LINE
===============================================================================

The engine of this bot is now:

- passive spread capture through pure market making
- maker-first combinatorial arbitrage on mutually exclusive outcomes
- zero-overhead latency arbitrage driven by free crypto spot and free Tree
  News data

The critical design constraint is also clear:

- if you do not have L2 book data, you do not have a valid backtest for the
  current pure market maker

The current operational corollary is also clear:

- if the forward paper deployment is not producing both snapshots and deltas in
  data/raw_ticks, you are not collecting a defensible pure-MM optimization
  dataset for next week's WFO

- once structural edge is established in WFO, position size is currently meant
  to be set as a separate risk step using scripts/kelly_sizer.py rather than
  treated as an unconstrained search parameter
- once a wide-tier WFO sweep completes, scripts/visualize_wfo_surface.py is the
  intended offline review surface for identifying broad profitable plateaus
  before any parameter choice is promoted into operator workflow

That is the system as it exists now.
