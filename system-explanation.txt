Polymarket Bot System Explanation

This file explains the system that actually runs on the VPS today.

The current deployment is a dual-strategy PAPER fund built from two continuous
background services:

- Sword, the grouped-event CLOB arbitrage loop.
- Shield, the favorite-longshot underwriting loop.

Both services consume live market data. Both launchers currently execute in
PAPER mode. Both services are independent and keep running in the background
until systemd stops them.

1. What Runs Continuously

The runtime center is `scripts/vps_master_scheduler.sh`.

That script owns the continuous loop behavior for both strategies. It is not a
timer, not a cron entry, and not a one-shot deploy helper. It is the thing that
keeps each pipeline alive.

The scheduler can run three modes:

- sword
- shield
- all

The deployed VPS uses one service per mode instead of one process running both
loops together. That matters because a Sword restart does not have to interrupt
Shield, and a Shield failure alert does not stop Sword from continuing to scan.

2. The Sword Service

Sword is the arbitrage lane.

Its systemd unit is `scripts/polymarket-sword-paper.service`. That unit starts
the scheduler with `--pipeline sword` and sets the loop interval to `300`
seconds.

Each Sword cycle is:

1. scan the live Gamma universe for grouped events that are still tradeable and
   have live order books;
2. keep only grouped events with at least three active outcomes;
3. require negative-risk mechanics at the event and leg level;
4. reject cumulative threshold ladders that are not true Dutch-book baskets;
5. pull the live YES book for every candidate leg from the CLOB;
6. decide whether the strip is executable as either:
   - a `BUY_YES_STRIP`, where the sum of asks is below `1 - fee_buffer`, or
   - a `SELL_NO_STRIP`, where the sum of bids is above `1 + fee_buffer`;
7. require every leg to clear the minimum displayed depth check;
8. write the surviving strip list to `config/live_executable_strips.json`.

Once targets exist, the launcher takes over.

3. How Sword Sizes Risk

Sword does not use a fixed strip size.

`scripts/launch_clob_arb.py` sizes every strip from the weakest displayed leg.

The important rule is simple: if one leg is only safe for ten displayed dollars
at the top of book, the entire basket is sized from that ten-dollar leg.

The launcher turns the scanner output into a strip cap by:

1. reading `min_leg_depth_usd_observed`;
2. rounding it down to a whole-dollar cap;
3. dividing by the strip execution price sum;
4. clamping again by the smallest displayed share size across the basket;
5. enforcing exchange minimum share and notional rules.

That is how Sword stays executable as a basket instead of pretending the
deepest leg defines safe size.

4. How Sword Fires Orders

The scheduler runs Sword in PAPER mode, but it still uses the live launcher.

The launcher signs every leg, builds the order payloads, and then sends all leg
submissions concurrently. The default time-in-force is `FOK`, so the deployed
service currently attempts each strip as a fill-or-kill basket.

The machine-readable receipt for each cycle goes to
`data/clob_arb_launch_summary_paper.json`.

The human-readable markdown receipt goes to `docs/clob_arb_receipt.md`.

5. Sword Anti-Legging Safety

Sword has an explicit anti-legging safety handler for partial execution.

This logic matters even though the VPS is in PAPER mode today, because the
launcher is wired the same way it would be in live trading.

The handler works like this:

1. If every leg fills completely, the strip is done.
2. If nothing fills, the strip is recorded as no fill or submitted.
3. If some legs fill and others do not, the launcher immediately treats the
   filled pieces as unwanted inventory and starts a flatten workflow.

Flatten Stage 1:

- submit IOC cleanup at the live best bid or best ask, depending on which side
  must be unwound.

Flatten Stage 2:

- if Stage 1 leaves inventory behind, reprice more aggressively by the larger
  of five cents or five percent from the Stage 1 price;
- clip the new price into the safe venue range;
- submit another IOC cleanup order.

Every flatten attempt is written into the Sword receipt, including whether the
launcher had to escalate to Stage 2 and whether any inventory still remained.

6. The Shield Service

Shield is the underwriting lane.

Its systemd unit is `scripts/polymarket-shield-paper.service`. That unit starts
the scheduler with `--pipeline shield` and sets the loop interval to `10800`
seconds.

Each Shield cycle is:

1. page through the live Gamma market set;
2. keep only active markets that are accepting orders and have an enabled order
   book;
3. resolve YES and NO token ids;
4. prefilter Gamma rows whose YES outcome price is already too expensive;
5. pull the live YES book from the CLOB;
6. choose the best live YES reference price;
7. keep only markets whose YES price is genuinely below five cents;
8. rank the surviving targets by volume, then liquidity, then cheaper entry;
9. cut the list down to at most `100` names;
10. write the target file to `data/flb_results_live.json`.

The `100` cap is hard-coded into the scanner defaults. It is not an operator
convention. The scanner enforces it directly.

7. How Shield Underwrites Risk

`scripts/launch_underwriter.py` turns the Shield target file into passive NO
quotes.

The current implementation expresses its Kelly allocation as a fixed
per-condition cap:

- `50` USD notional per market;
- price `0.95` on the NO side;
- share size computed as `50 / 0.95` and quantized to exchange precision.

Every submitted order is:

- a limit order;
- GTC;
- post-only;
- one order per selected condition.

The current continuous deployment is PAPER, so the transport intercepts the
fully signed payload locally. The launcher still validates credentials and still
builds the same payload shape it would use in live trading.

Shield writes:

- `data/underwriter_launch_summary_paper.json`
- `docs/underwriter_launch_report_paper.md`

In live mode only, Shield also writes a state file so it can skip duplicate
resting orders on later runs. In paper mode that state write is skipped.

8. How Reporting Works

The reporting contract is file-first.

The scanner writes the target file.
The launcher writes the JSON summary and the markdown report.
The scheduler then reads the JSON summary and calls
`scripts/send_strategy_telegram_alert.py`.

That alert bridge sends:

- a Shield summary with active targets, staged orders, rejected orders, top
  categories, and sample market names;
- a Sword summary with executable strip count, grouped event count, status
  counts, and top strip previews;
- failure alerts with strategy, failing stage, and message text.

There is also a strict communication-check mode, which is used by the VPS smoke
test to verify that Telegram is still reachable.

9. How The OS Keeps It Alive

The active paper services use standard systemd resilience settings:

- `Restart=always`
- journald logging
- `NoNewPrivileges=true`
- raised file descriptor limits

They also create `/dev/shm/secrets`, decrypt `.env.age` into that memory-backed
directory during startup, and remove the decrypted file on shutdown.

The checked-in log rotation policy is `config/polymarket-bot.logrotate`. It
rotates live `.log` and `.jsonl` files daily, keeps `14` rotations, compresses
old files, and uses `copytruncate` with a `100M` max-size threshold.

The checked-in `MemoryMax=750M` systemd hard cap lives in
`config/polymarket-master.service`, which is the repo's hardened master-service
template. The active paper units are separate files focused on running one
pipeline each, while the memory-cap pattern is captured in that master-service
scaffold.

10. What A New Developer Should Remember

If you need the shortest correct mental model, it is this:

- Sword continuously hunts grouped CLOB Dutch books.
- Shield continuously hunts sub-5c YES longshots and underwrites them with NO
  quotes.
- The scheduler owns the loops.
- The services keep the loops alive.
- The launchers write the receipts.
- Telegram is driven from those receipts.
- Sword manages partial-fill risk with a two-stage flatten handler.
- Shield manages portfolio breadth by ranking and hard-capping the target set at
  `100` names.

That is the current deployed system.
