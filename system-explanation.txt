1. CURRENT REPOSITORY REALITY
===============================================================================

The checked-in repository is a hybrid, transitional trading system.

It is not accurate to describe the repo as only a legacy directional bot.
It is also not accurate to describe it as already having fully retired the
directional stack from live execution.

The checked-in code currently contains all of the following:

1. Bar-close directional entry logic centered on PanicDetector
2. Resolution Probability Engine logic, including fast-strike support
3. Pure market making logic
4. Optional SI-8 oracle/news latency arbitrage infrastructure
5. Optional SI-9 combinatorial arbitrage infrastructure

The strongest architectural truth in the current checkout is:

- src/bot.py still evaluates a bar-close panic path and can open positions from
  that path
- src/strategies/pure_market_maker.py is real, wired, and scheduled when
  PURE_MM_ENABLED is true
- oracle and combo-arb paths exist, but they are feature-flagged and not on by
  default in src/core/config.py

That means the repo should be treated as a mixed execution stack, not as a
single-strategy engine.

3. DATA PLANE AND LOAD SHEDDING
===============================================================================

The repo is strongly L2-aware.

The checked-in code does all of the following:

- ingests Polymarket market WebSocket traffic
- ingests L2 book traffic
- maintains order book trackers and L2 books
- gates decisions on L2 reliability
- uses a separate L2-active subset for heavier book work

Important distinction:

- max_active_l2_markets governs the top fully L2-active subset
- that is not the same thing as proving the entire runtime only tracks 25
  markets everywhere in every deployment

Current checked-in code behavior at startup:

- src/bot.py sorts discovered markets by volume
- src/bot.py slices self._markets to max_active_l2_markets
- src/bot.py also builds _l2_active_set from the top l2_limit markets across
  all wired markets

Current checked-in config defaults:

- max_active_l2_markets = 25
- pure_mm_max_markets = 25

But the latest saved runtime artifact disagrees with the current source:

- logs/bot.jsonl records markets_selected = 174
- that same log records load_shedding_applied with l2_active = 25

That is the clearest sign that the saved runtime logs do not come from the
exact same code/config state as the current checkout.

Therefore the accurate documentation stance is:

- the checked-in source is designed to cap the heavy L2-active subset at 25
- the last saved runtime artifacts show 25 L2-active markets but a larger
  broader market set was still selected in that run
- do not state "the tracked market universe is hard-capped at 25 markets"
  without qualifying whether you mean source design or a specific deployment
