Polymarket Bot System Explanation

This file describes the runtime that actually governs trading now.

It does not describe deleted adapter lore.
It does not describe optional historical code as if it were live.
It describes the Level 0 kernel that currently owns risk.

1. Process Model

The runtime is still centered on src/bot.py.

The bot starts and owns:

- websocket and L2 consumers,
- trade and book processing loops,
- timeout, stats, cleanup, health, and paper-summary loops,
- the orchestrator tick loop,
- the wallet-balance poll loop when the live boundary exposes a balance
  provider,
- reward-sidecar maintenance when the reward lane is enabled,
- frozen legacy loops only when their lane states are explicitly turned back
  on.

This is still one process with shared in-memory state. It is not a set of
independent daemons.

2. Execution Boundary

src/execution/live_execution_boundary.py is still the only legitimate venue
boundary.

In PAPER mode it is intentionally empty.

In LIVE mode it builds the venue-facing objects:

- transport,
- adapter,
- nonce manager,
- signer,
- wallet-balance provider,
- OFI exit router.

No lane should create a separate venue-facing execution stack outside this
boundary.

3. Orchestrator Scope

The current orchestration layer is src/execution/multi_signal_orchestrator.py.

It is not the old giant adapter router.

The current orchestrator owns:

- DispatchGuard,
- PriorityDispatcher,
- guard observability,
- OFI paper accounting,
- reward poster adapter,
- optional load shedding,
- optional wallet-balance plumbing,
- optional OFI exit routing.

It no longer defines the system around SignalCoordinationBus, OfiSignalBridge,
CtfPaperAdapter, or Si9PaperAdapter. Do not use that model.

4. Unified Pre-Dispatch Gate

src/execution/priority_dispatcher.py is the single admission bottleneck.

Before a dispatcher-backed intent reaches the venue path, it goes through:

1. bound pre-dispatch gates,
2. shared DispatchGuard checks,
3. MEV envelope construction and serialization,
4. paper, dry-run, or live execution,
5. normalized DispatchReceipt creation.

That is the current execution contract.

The point of the reset was to stop scattering entry admission across many
strategy-specific adapters.

5. Health States

src/execution/orchestrator_health_monitor.py now exposes three effective
states:

- GREEN
- YELLOW
- RED

Their operational meaning is:

- GREEN: normal operation.
- YELLOW: degraded-risk mode. Position management is still allowed. New live
  entries are not.
- RED: hard stop.

This is not only advisory telemetry.

The monitor binds into PriorityDispatcher and rejects new dispatcher-backed
entry sources while YELLOW. It also exposes higher-level fields such as
allows_position_management and allows_new_panic_entries so the bot can block
new directional risk above the dispatcher too.

If the runtime is YELLOW, the correct behavior is to manage or flatten risk,
not add risk.

6. Lane States

Current default lane states in src/core/config.py are:

- panic = LIVE
- contagion = SHADOW
- ofi_momentum = OFF
- rpe = OFF
- oracle = OFF
- pure_market_maker = OFF
- si9_combo = OFF
- cross_market = OFF
- si10_bayesian = OFF
- reward = OFF

The practical outcome is simple:

- Panic is the only live directional candidate.
- Contagion is the active shadow incubator.
- Pure MM, SI-9, standard OFI, Oracle, and the rest are frozen.
- Reward infrastructure exists in code, but it is not a default always-on
  lane.

7. Panic

Panic is the runtime's only live directional candidate.

It is still bot-managed through PositionManager rather than through a dedicated
dispatcher execution adapter.

What actively governs it now:

- lane-state gating,
- tradeability checks,
- cooldowns,
- book-quality gates,
- ensemble-risk vetoes,
- bot-level health gating.

If you are debugging live directional behavior, start with Panic first.

8. Contagion

Contagion is no longer supposed to be understood as a special fast-strike bypass.

Current behavior:

- the detector still evaluates on BBO updates inside src/bot.py,
- shadow contagion uses dispatcher-backed admission in PAPER mode before it
  opens a shadow position,
- live contagion evaluates through the unified pre-dispatch gate before it
  hands off to the existing position-management fast-strike machinery.

So the accurate statement is not "contagion became a full orchestrator lane".
The accurate statement is "contagion no longer bypasses unified admission".

9. Reward Poster Sidecar

The repository now contains a reward-posting subsystem.

The main runtime object is src/rewards/reward_poster_sidecar.py.

It does all of the following:

- select reward markets,
- maintain working maker quotes,
- cancel stale or unsafe quotes,
- track one-sided fills and local inventory,
- emit reward shadow persistence payloads.

It does not own its own venue adapter.

The sidecar uses the shared dispatcher through RewardPosterIntent and
RewardExecutionHints.

The reward execution contract is strict:

- post_only must be True,
- time_in_force must be GTC,
- liquidity_intent must be MAKER_REWARD,
- taker escalation is disabled.

That means reward posting is intentionally maker-only and centrally gated.

10. Reward Shadow Attribution

Reward measurement does not expand the SQL schema.

Instead, src/monitoring/trade_store.py now allows shadow persistence to accept
extra_payload and stores it only in payload_json.extra_payload inside the
journaled payload.

That payload can include fields such as:

- reward_to_competition,
- reward_daily_usd,
- reward_max_spread_cents,
- queue_residency_seconds,
- fill_occurred,
- estimated_reward_capture_usd,
- estimated_net_edge_usd,
- quote_id,
- quote_reason,
- emergency_flatten.

This is deliberate. Measurement data belongs in the journal payload, not in an
ever-growing SQL table.

11. Persistence Contract

src/monitoring/trade_store.py now uses a journal-first persistence contract for
both closed trades and shadow trades.

The write sequence is:

1. write the trade_persistence_journal row,
2. begin the ledger transaction,
3. write the ledger row,
4. mark the journal row RECORDED on success,
5. mark the journal row FAILED on error.

The journal tracks explicit states:

- PENDING
- RECORDED
- FAILED

Persistence failures are now supposed to be loud and attributable. Silent drops
are treated as a bug.

12. WAL-Safe Measurement

The trade store also owns WAL-safe remeasurement snapshots.

The snapshot process does not rely on a naive copy of the live SQLite file.
It does:

- a passive WAL checkpoint,
- an SQLite backup into a snapshot database,
- optional JSONL export of the persistence journal,
- a manifest JSON that records paths and accounting status.

That manifest is the measurement anchor for offline reporting.

13. What Actually Governs The Bot

If you strip away dead architecture language, the current governing logic is:

- Panic is the only live candidate.
- Contagion is the shadow incubator.
- Reward posting uses the shared dispatcher and shared persistence surfaces.
- PriorityDispatcher is the front door for dispatcher-backed risk.
- YELLOW means exits and flattening only.
- trade_store is journal-first and loud on failure.

Everything else should be treated as frozen, legacy, or secondary until it is
explicitly reactivated.
