===============================================================================
POLYMARKET BOT - SYSTEM EXPLANATION
Last updated: March 19, 2026
===============================================================================

This repository is no longer a bar-driven directional trading bot.

The live system is now a high-frequency pure market maker and latency
arbitrage engine for Polymarket's CLOB. Any reader, operator, or future AI
agent should treat that as the primary architectural truth.


1. CURRENT REALITY
===============================================================================

The active production architecture has three live strategy families:

1. Pure Market Maker
2. SI-9 Combinatorial Arbitrage
3. Latency Arbitrage
   - SI-7 crypto fast-strike
   - SI-8 news/oracle fast-strike

The old directional stack is retired for live trading.

Retired from live architecture:
- PanicDetector-driven entries
- Mean-reversion drift entries
- bar-entry logic as the primary execution path
- deployment decisions based on legacy z-score or drift-ceiling style knobs

Some of that code still exists for research, compatibility, and legacy replay,
but it is not the live engine. If a document, script, or config discussion
frames this repo as a panic-fading or bar-signal bot, that description is
wrong.


2. EXECUTIVE SUMMARY
===============================================================================

The bot is a Polymarket microstructure system that does two things well:

- Rest passive liquidity where the spread is worth capturing.
- Hit stale liquidity when an external information edge appears before the
  CLOB reprices.

In practice that means:

- full L2 order book reconstruction on the top active markets
- passive NO-token quoting for spread capture
- maker-first execution for mutually exclusive event arbitrage
- taker fast-strike execution when crypto spot or breaking news moves faster
  than the Polymarket book
- strict latency, liquidity, and toxicity gates around every entry path


3. HIGH-LEVEL RUNTIME ARCHITECTURE
===============================================================================

3.1 Data plane

The live bot consumes and maintains:

- Polymarket trade and book streams
- dedicated L2 snapshot and delta data
- optional external price and oracle feeds for latency arbitrage

The system load-sheds aggressively:

- only the top L2 markets by volume receive full depth reconstruction
- the remaining markets can be tracked in lighter-weight modes

L2 processing is first-class infrastructure, not an optional add-on.
Worker processes can reconstruct books off the main asyncio loop and publish
state back through shared memory so strategy code reads current BBO and depth
without blocking the bot.

3.2 Strategy plane

The strategy layer is centered on three active paths:

- PureMarketMaker in src/strategies/pure_market_maker.py
- SI-9 combo arb detection and execution routing
- SI-7 and SI-8 latency-arb routes that flow through the fast-strike taker
  path in the position manager

3.3 Execution and risk plane

Execution is split by intent:

- passive quoting and maker routing for spread capture
- immediate taker routing for stale liquidity sniping
- stateful position management, stop logic, and adverse-selection defenses

Risk enforcement is continuous:

- latency guard
- L2 reliability checks
- adverse-selection fast-kill logic
- inventory caps for market making
- wallet and exposure caps

3.4 Monitoring and recovery

The system records fills and checkpoints, logs structured events, sends
Telegram alerts, and relies on circuit breakers around hot loops. If a core
loop repeatedly fails, the bot suspends and resets instead of continuing in a
half-broken state.


4. ACTIVE STRATEGY 1 - PURE MARKET MAKER
===============================================================================

This is the primary live engine.

What it does:

- selects the highest-volume L2 markets
- quotes the NO token passively
- posts a best-bid-style passive buy quote
- only posts the ask when inventory already exists
- cycles inventory from filled bid to passive ask to capture spread

Important implementation details:

- quoting is inventory-aware, so the engine does not rely on naked asks
- quote size is capped by a per-market-maker inventory budget
- selected markets come from the active L2 universe, not from bar signals
- the engine runs in a tight loop suitable for HFT quote maintenance

Why it can make money:

- the bot earns spread when a passive bid fills and that inventory is later
  distributed through a passive ask
- the strategy is designed for high-liquidity binary books where queue
  position, spread width, and quote survival matter more than slow-moving
  directional forecasts

Why it can avoid getting run over:

The bot does not leave passive quotes resting blindly. It cancels when the
microstructure turns toxic. The current live design explicitly uses:

- OFI, or order flow imbalance
- depth evaporation over a short time window
- L2 reliability gates
- latency health and fast-kill conditions
- optional iceberg and venue-microstructure cues from the broader system

The key idea is simple: capture passive spread when flow is benign, and cancel
before informed flow runs through the quote.


5. ACTIVE STRATEGY 2 - SI-9 COMBINATORIAL ARBITRAGE
===============================================================================

SI-9 is a maker-first state machine for mutually exclusive market clusters.

When a Polymarket event exposes a set of mutually exclusive outcomes and the
cluster is marked negRisk-compatible, the bot checks whether the combined YES
pricing is inconsistent with a guaranteed payout structure.

Core idea:

- if mutually exclusive outcomes are mispriced, the bot can buy a balanced set
  of legs whose combined settlement value is locked in

Current live design:

- clusters are built from mutually exclusive event groups
- the hardest leg to fill is chosen as the maker leg
- that bottleneck leg is worked passively first
- the remaining legs are routed after the maker-first state machine confirms
  the opportunity can be completed defensibly

This is not a generic stat-arb module. It is a routing-aware execution system
for Dutch-book style mispricings in mutually exclusive Polymarket structures.


6. ACTIVE STRATEGY 3 - LATENCY ARBITRAGE
===============================================================================

Latency arbitrage is the taker side of the architecture. When an external
source of truth moves faster than the Polymarket book, the bot attempts to hit
stale liquidity immediately.

There are two live latency-arb flavors.

6.1 SI-7 - Crypto Fast-Strike

SI-7 targets crypto-linked prediction markets.

What drives it:

- free external crypto spot feeds for BTC and ETH
- the Resolution Probability Engine in src/signals/resolution_probability.py
- Black-Scholes-style threshold pricing for crypto strike markets
- a fast retrigger loop that re-evaluates crypto markets whenever spot moves
  enough to create a fresh divergence

How it executes:

- when the model detects a sufficiently stale Polymarket price and latency is
  healthy, the position manager can use the fast-strike taker path
- those positions intentionally skip passive chasing because the edge is the
  speed advantage itself

Operational meaning:

- SI-7 is not slow research alpha
- it is an HFT stale-order sniping path driven by free crypto market data

6.2 SI-8 - News Arbitrage

SI-8 is the generalized oracle/news fast-strike path.

Current live deployment focus:

- Tree News WebSocket firehose for breaking news and resolution-relevant event
  updates

How it works:

- off-chain adapters emit OracleSnapshot objects
- OracleSignalEngine converts verified external state changes into a market
  probability view
- if Polymarket is stale and spread and confidence gates pass, the bot routes
  the trade into the fast-strike execution path

Zero-overhead architecture rule:

- paid non-crypto and paid sports feeds are hard-disabled in the intended live
  deployment model
- the repo still contains adapter scaffolding for sports and Odds API style
  integrations, but those are not part of the target production architecture
- the production direction is zero-overhead latency arb built on free crypto
  spot data and free Tree News WebSocket data

If you are configuring live SI-8 today, the intended production assumption is:

- Tree News is active
- paid The Odds API style paths are not part of the zero-overhead live stack


7. DATA AND MARKET MICROSTRUCTURE REQUIREMENTS
===============================================================================

The architecture pivot changed the data requirements completely.

Trade receipts alone are no longer enough to understand or simulate the live
bot.

The current live system depends on L2 information for:

- queue-aware passive quoting
- best bid and best ask evolution
- OFI calculation
- depth-evaporation detection
- maker fill realism
- stale-book and toxicity gating

For the current bot, L2 is not optional historical enrichment. It is core
strategy data.


8. BACKTEST AND WFO PIVOT
===============================================================================

Walk-Forward Optimization has been realigned around the new pure market maker
architecture.

Important facts:

- WFO now supports a pure market maker replay adapter
- that replay path depends on L2 snapshots and L2 deltas
- pure market maker WFO is invalid on trade-only datasets

Why trade-only data is insufficient:

- passive maker fills cannot be simulated defensibly from trade receipts alone
- OFI cannot be reconstructed from trade-only data
- depth evaporation cannot be measured from trade-only data
- quote cancellation logic cannot be replayed faithfully without book changes

Current enforced behavior:

- when WFO is run with strategy_adapter set to pure_market_maker and the
  replay sees zero L2 book updates, the optimizer logs a missing-L2 warning
  and returns no usable result

This is deliberate. A fake pure-MM backtest on trade-only data is worse than
no backtest because it produces parameter outputs that look scientific but do
not reflect the live execution problem.

If you want valid pure market maker backtests, your dataset must contain:

- L2 snapshots
- L2 deltas
- correctly ordered book events over time

Trade-only archives may still be useful for limited legacy research, but they
are not sufficient for the current live pure market making architecture.
