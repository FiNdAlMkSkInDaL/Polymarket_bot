CHANGE REPORT — Exception Circuit Breaker for Critical Loops
Date: 2026-03-05
================================================================================

BACKGROUND
----------
During the March 5th crash-hardening sweep, broad `except Exception` blocks
were introduced into `_process_trades` (bot.py) and `_decision_loop`
(adverse_selection_guard.py).  While this prevented isolated errors from
permanently killing concurrent tasks, swallowing every exception risked
masking severe L2 state corruption or fast-kill evaluation errors — leading
to degraded and dangerous trading states operating silently.

This refactoring replaces the blanket exception handlers with:
  1. Targeted catches for expected, non-fatal exceptions.
  2. An "Exception Circuit Breaker" that trips after a configurable
     threshold of unexpected errors and triggers the bot's graceful
     shutdown sequence.

================================================================================
FILES MODIFIED
================================================================================

1. src/core/exception_circuit_breaker.py  (NEW)
   - New module: ExceptionCircuitBreaker class.
   - Rolling-window error counter using collections.deque of timestamps.
   - record() returns True when threshold breached (breaker trips).
   - Once tripped, always returns True (latching behaviour).
   - Configurable: threshold (default 5), window_s (default 60.0).
   - Uses time.monotonic() for reliable monotonic timing.

2. src/bot.py
   - Added import: ExceptionCircuitBreaker from src.core.exception_circuit_breaker.
   - Added _trade_loop_breaker instance (threshold=5, window_s=60.0) in __init__.
   - _process_trades inner except block refactored:
     a. KeyError caught explicitly — logs as warning, continues (stale data).
     b. Remaining Exception: logs error, records in circuit breaker.
        If breaker trips → logs CRITICAL, sends Telegram alert, schedules
        self.stop() via asyncio.ensure_future(), returns from loop.
   - AdverseSelectionGuard construction: added on_shutdown=self.stop kwarg.

3. src/signals/adverse_selection_guard.py
   - Added import: ExceptionCircuitBreaker from src.core.exception_circuit_breaker.
   - __init__ signature extended with on_shutdown: Callable[[], Any] | None = None.
   - Added _loop_breaker instance (threshold=5, window_s=60.0) in __init__.
   - _decision_loop except block refactored:
     a. (KeyError, AttributeError) caught explicitly — logs as warning,
        continues (stale/missing book data, non-fatal).
     b. Remaining Exception: logs error, records in circuit breaker.
        If breaker trips → logs CRITICAL, sends Telegram alert, sets
        _running=False, schedules on_shutdown() callback, returns from loop.

================================================================================
CIRCUIT BREAKER LOGIC
================================================================================

Threshold: 5 unexpected exceptions within a 60-second rolling window.

Mechanism:
  - Each unexpected (non-targeted) exception appends time.monotonic() to a
    deque.  Timestamps older than window_s are evicted on each call.
  - If len(timestamps) >= threshold → breaker trips (latching).
  - Once tripped, the loop exits and triggers graceful shutdown:
    * Logs CRITICAL with error count.
    * Sends Telegram alert.
    * Calls self.stop() (TradingBot) — cancels all orders, cancels chasers,
      stops all subsystems, calls trade_store.clear_live_state().

Targeted (non-fatal) exceptions that DO NOT count towards the breaker:
  - _process_trades: KeyError (stale asset data in shared dicts).
  - _decision_loop: KeyError, AttributeError (missing book/asset data).
  - Both loops: asyncio.CancelledError (always re-raised, never caught).
  - Both loops: asyncio.TimeoutError on queue.get() (explicit continue).

CHANGE REPORT — SI-3 Cross-Market Signal: Logging Payload & Direction Fix
Date: 2026-03-05
================================================================================

BACKGROUND
----------
A 20-hour production log analysis via parse_si3_logs.py revealed two critical
defects in the Cross-Market Stat-Arb (SI-3) shadow signal system
(src/signals/cross_market.py):

1. DATA DISCONNECT — The [SHADOW] cross_market_signal structlog payload only
   emitted truncated condition_ids (lagging, leading). It lacked the specific
   YES/NO token asset_ids required to join log entries to tick data for
   offline P&L reconstruction.

2. DIRECTIONAL BUG — 100% of the 35 logged signals over 20 hours carried
   direction="YES". For a stat-arb pair trade across diverse correlated
   markets, a mixed YES/NO distribution is expected. Root cause: a
   copy-paste error in the direction assignment.

================================================================================
MATHEMATICAL FLAW — DIRECTION CALCULATION
================================================================================

The original code determined the lagging/leading market from the z-score sign
but hardcoded direction="YES" in BOTH branches:

    if z > 0:
        lagging_id = mkt_b
        leading_id = mkt_a
        direction = "YES"          # ← correct only when leader went UP
    else:
        lagging_id = mkt_a
        leading_id = mkt_b
        direction = "YES"          # ← BUG: copy-paste of the z > 0 case

The z-score tells us WHICH market is lagging, but the DIRECTION of the
convergence trade depends on the leader's return sign:

  - If the leader's YES-token return is positive (probability increased),
    the lagging market should converge UPWARD  → buy YES on the lagger.
  - If the leader's YES-token return is negative (probability decreased),
    the lagging market should converge DOWNWARD → buy NO on the lagger.

CORRECTED LOGIC:

    if z > 0:
        lagging_id = mkt_b
        leading_id = mkt_a
        leader_return = ret_a
    else:
        lagging_id = mkt_a
        leading_id = mkt_b
        leader_return = ret_b

    direction = "YES" if leader_return > 0 else "NO"

This produces the expected statistical mix of YES/NO signals that tracks
the actual directional co-movement of correlated prediction markets.

CHANGE REPORT — Post-Stop-Loss Market Cooldown (Re-entry Bleed Fix)
Date: 2026-03-05
================================================================================

BACKGROUND
----------
Production logs on 2026-03-05 showed the bot repeatedly re-entering the
same market (0x561cd8d035bac38ed04e23d7882a126da38d7e) after successive
stop-losses, burning through POS-2 and POS-3 within a two-minute window:

  20:11  Entry Filled — POS-2, Buy NO @ 0.33¢, Target: 0.37¢
  20:11  Position Closed — POS-2, PnL: -6.98¢ (stop_loss)
  20:11  Position Closed — POS-2, PnL: -2.60¢ (target)
  20:11  Entry Filled — POS-3, Buy NO @ 0.33¢, Target: 0.37¢
  20:12  Position Closed — POS-3, PnL: -7.14¢ (stop_loss)
  20:12  Position Closed — POS-3, PnL: -4.27¢ (target)

Two compounding bugs caused this:

1. NO POST-STOP-LOSS COOLDOWN PER MARKET
   When a stop-loss fires, on_exit_filled() immediately sets the position
   state to CLOSED. The per-market concentration check in open_position()
   queries get_open_positions(), which only counts positions in states
   ENTRY_PENDING / ENTRY_FILLED / EXIT_PENDING. A CLOSED position is
   invisible to this check, so the next signal evaluation sees zero open
   positions on the market and can open a fresh entry immediately.

   The existing signal_cooldown_minutes = 0.5 (30 seconds) starts
   counting from signal fire time — not from close time. By the time
   a stop-loss fills ~60-90 seconds later, the 30-second cooldown has
   already expired, leaving a completely open re-entry window.

2. PANIC SIGNALS LACK A PER-MARKET SOURCE COOLDOWN
   Drift signals have an explicit _drift_cooldowns dict enforcing a
   60-second per-market gap between consecutive drift signal evaluations.
   Panic signals have NO equivalent per-source cooldown. On every bar
   after a stop-loss, if the detector's z-score threshold is met, a new
   panic signal fires immediately regardless of recent trade history on
   that market.

================================================================================
ROOT CAUSE TRACE
================================================================================

Stop-loss → re-entry cycle (per bar):

  1. PanicDetector fires. lifecycle.record_signal() stamps t=0.
  2. Position opens (ENTRY_PENDING → ENTRY_FILLED → EXIT_PENDING).
  3. Price moves adversely. StopLossMonitor._trigger_stop() calls
     force_stop_loss(pos) → on_exit_filled(pos, reason="stop_loss").
  4. Position state = CLOSED. No per-market cooldown recorded.
  5. Next bar: is_cooled_down() checks (now - t=0) ≥ 30s → TRUE (it has
     been > 30s since the signal, not since the close).
  6. per_market_limit check: get_open_positions() returns [] for this
     market (position is CLOSED). Check passes.
  7. New position opens on the same losing market. Repeat from step 3.

FILES MODIFIED
================================================================================

1. src/core/config.py
   - Added new field to StrategyParams immediately after stop_loss_cents:
       stop_loss_cooldown_s: float = _env_float("STOP_LOSS_COOLDOWN_S", 300.0)

DESIGN DECISIONS
================================================================================

Cooldown scope: market_id (condition_id), not asset_id.
  The stop is keyed to the prediction-market condition, not the specific
  YES or NO token, so that both panic (BUY_NO) and RPE (BUY_YES) signal
  paths are blocked from re-entering the same market concept.

Source-level enforcement (bot.py) over risk-gate enforcement (position_manager):
  The cooldown is enforced at the signal evaluation stage so that the
  position manager's risk gate remains a pure structural guard
  (max_positions, exposure, circuit-breaker). Mixing a time-based
  post-loss heuristic into the structural gate would broaden its
  responsibility and complicate unit testing.

Default 300 seconds (5 minutes):
  Chosen to cover:
    - 5 × 1-minute bars (typical signal evaluation cadence)
    - Latency recovery and WS reconnect window (up to ~2 minutes)
    - Mean-reversion half-life estimate (~1-2 min) plus 3× safety margin
  Tunable downward (e.g., STOP_LOSS_COOLDOWN_S=120) if live data shows
  excessive missed opportunities on fast-recovering markets.

No cooldown on "target" or "timeout" exits:
  Only stop_loss reason triggers the 300s blackout. Positions closed at
  target or via timeout indicate the market environment was acceptable;
  re-entry on the next signal is legitimate.

CHANGE REPORT — Multi-Core Distributed Execution Architecture
Date: 2026-03-05
================================================================================

BACKGROUND
----------
The bot previously ran as a single asyncio process.  Under full load (130+
markets), the L2 order-book reconstruction WebSocket callbacks and the
PCE/SI-3 O(N²) correlation matrix math competed for the same event loop
with order routing and risk management, introducing measurable latency
jitter on the execution path.

This refactoring transitions the bot to a multi-core process topology:

  Process 0  (main)   — asyncio event loop for CLOB order routing,
                        risk gates, signal evaluation, monitoring.
  Processes 1–N (L2   — one asyncio+uvloop process per shard of markets;
             workers)   each maintains L2WebSocket + L2OrderBook instances
                        for its shard and publishes BBO updates via
                        shared memory.
  Process N+1 (PCE)   — owns PortfolioCorrelationEngine +
                        CrossMarketSignalGenerator; receives bar-return
                        feeds from main process and publishes results
                        and VaR gate responses.

All existing safety pillars (heartbeat, latency guard, circuit breaker,
stop-loss cooldown, re-entry lock) are preserved and extended to span
cross-process boundaries.  Single-process mode remains available as a
fallback (MULTICORE_ENABLED=false).

================================================================================
CHANGE REPORT — IPC & Worker Lifecycle Hardening
Date: 2026-03-05
================================================================================

BACKGROUND
----------
The multi-core refactor successfully bypassed the GIL and eliminated
CPU-bound latency jitter.  Four systemic risks remained in the distributed
process topology:

1. SPIN-LOCK DATA CORRUPTION — SharedBookReader spun 100 iterations on the
   write-lock byte.  If the L2 worker was preempted mid-write, the reader
   exhausted the cap and proceeded to unpack a half-written struct, silently
   returning torn BBO / depth data to the AdverseSelectionGuard hot path.

2. QUEUE DROP OPACITY — All put_nowait() calls across six queues were wrapped
   in bare `except Exception: pass` blocks.  queue.Full exceptions on a
   stalled worker were completely invisible — no counter, no log, no signal
   that messages were being silently discarded.

3. SIGKILL SHARED-MEMORY LEAK — Python's cleanup_shm() runs during
   bot.stop() → process_manager.stop_all().  An OS OOM-killer sending
   SIGKILL to the main process bypasses all Python atexit / signal handlers,
   leaving pmb_<hash> blocks stranded in /dev/shm indefinitely.  The next
   process boot then fails with "file exists" on shm allocation.

4. ZOMBIE WORKERS — With daemon=False, child processes (l2_worker,
   pce_worker) survive SIGKILL of the main process indefinitely, holding
   open WebSocket connections and consuming CPU.  No parent-death detection
   existed in the worker heartbeat loops.

================================================================================

1. src/core/ipc.py
   ─────────────────
   a) Added IPCReadError(Exception) class at module level.
      Raised when a shared-memory read cannot return consistent data

      Added two new fields:
        _last_snapshot: SharedBookSnapshot | None = None

        _spin_failures: int = 0

================================================================================
SPIN-LOCK FALLBACK POLICY
================================================================================

Return-cached-snapshot vs raise-IPCReadError:
  - On any read AFTER the first successful read: return _last_snapshot.
    The 50ms AdverseSelectionGuard hot path must never crash; a
    nanosecond-stale but structurally consistent snapshot is always
    preferable to an exception that kills the task.
  - On the FIRST read with no prior snapshot: raise IPCReadError.
    There is no safe fallback value for an uninitialized book; callers
    must handle initialization failures explicitly.

Rate-limited logging:
  Log every 100th spin failure (not every occurrence).  In a 50ms poll
  cycle, 100 consecutive lock contentions would represent ~5s of continuous
  write contention — genuinely anomalous and worth surfacing without
  flooding logs at tick frequency.

================================================================================
DAEMON FLAG & ORPHAN DETECTION INTERACTION
================================================================================

Both mechanisms target the same failure (zombie workers after parent death)
from different layers:

  daemon=True  — OS-level: parent death → SIGTERM to child (immediate).
                 Reliable for SIGKILL of parent; may not fire if parent
                 is merely hung.

  ppid check   — Python-level: detects hung/detached parent within one
                 heartbeat cycle (0.5s).  Triggers an orderly asyncio
                 shutdown (task cancellation, PCE state save, log flush)
                 rather than abrupt SIGTERM.  Works even when daemon
                 semantics are unavailable (e.g., inside some
                 container runtimes).

Neither mechanism alone is sufficient; both are required.

================================================================================
QUEUE DROP COUNTER CONVENTIONS
================================================================================

Hot-path queues (bbo_queue, pce_output_queue, var_response_queue):
  - Nonlocal int counter (closure-scoped inside the async worker function).
  - Log at "queue_full_drop" event level WARNING.
  - Rate: every 100th drop.  First drop (count==1) is always logged.
  - Counter persists for worker lifetime (monotonically increasing).

Control queues (add_asset / remove_asset):
  - Log every occurrence — these are rare administrative messages; a full
    control queue indicates a severely stalled worker.

VaR gate (position_manager):
  - Log every occurrence — fail-closed implications make every dropped
    VaR check operationally significant.

Main process (pce_input_queue in bot.py):
  - Instance counter _pce_queue_drops on the Bot object.
  - Same 100-drop log-rate policy.

CHANGE REPORT — WFO Zero-Trade Root Cause & DataLoader Backfill Repair
Date: 2026-03-08
================================================================================

BACKGROUND
----------
Walk-forward optimisation had been running for multiple days while producing
zero trades on every fold (`total_positions: 0`, `pnl: 0.0`) despite replaying
roughly 30,000 events per fold. The initial suspicion was that the
BotReplayAdapter gate funnel was overly strict (PanicDetector, EQS, Kelly, or
PCE), but the real failure occurred earlier: the strategy aggregators were not
forming 1-minute bars, so the signal path never meaningfully executed.

Root cause:
  Historical backfilled tick files stored `local_ts` near the time the
  backfill job ran, not the calendar day encoded in the directory path.
  DataLoader had a synthetic-timestamp repair, but it only activated when the
  ENTIRE file's timestamp span was < 60 seconds.

  The sampled backfilled files exhibited a multi-cluster pattern:
    - YES trades compressed into ~14 seconds
    - NO trades compressed into ~16 seconds
    - YES and NO clusters occurred at different backfill instants
    - overall file span ≈ 100 seconds

  Because the overall span exceeded 60s, the repair never triggered even
  though each asset individually had far too little elapsed time for the
  OHLCVAggregator to close 60-second bars. Result: zero bar closures, zero
  PanicDetector evaluations of consequence, and therefore zero entries.

CHANGE REPORT — 3-Stage WFO Optimization Pipeline
Date: 2026-03-08
================================================================================

BACKGROUND
----------
The --max-markets flag proved the discovery-sweep concept but exposed a
missing piece: the manual handoff between three logical phases
(discovery → refinement → full-universe validation) required running three
separate CLI commands, copying JSON files by hand, and computing narrowed
parameter bounds manually.  This change automates the entire pipeline as a
single orchestrator script with three phases:

  Phase 1 — Discovery Sweep:
    5 markets, 14 train days, 50 Optuna trials.
    Identifies a viable parameter region cheaply (~21× cheaper than full run).
    Output: logs/phase1_champion.json (24 best-trial params + fold metadata).

  Phase 2 — Refinement Sweep:
    10 markets, 30 train days, 100 trials.
    Initialises Optuna study with the Phase 1 champion as warm-start seed.
    Search space constrained to a ±15% neighbourhood around Phase 1 best params
    (computed by compute_narrowed_bounds()).
    Output: logs/phase2_champion.json, logs/phase2_bounds.json.

  Phase 3 — Full-Universe Validation:
    31 markets (full universe), 30 train days, 1 trial.
    Near-deterministic: bounds locked to epsilon neighbourhood of Phase 2 champion.
    No further optimisation — pure out-of-sample validation.
    Output: logs/final_validation_tearsheet.json, logs/phase3_locked_bounds.json.

Additionally, the dynamic search-space narrowing required modifications to
WFO itself so that Optuna can receive external per-parameter bounds at runtime.

================================================================================
SEARCH-SPACE NARROWING MATH
================================================================================

SEARCH_SPACE contains 24 parameters.  For Phase 2, the narrowing formula is:

  margin = 0.15 × (orig_hi − orig_lo)          (15% of full domain width)
  new_lo = max(orig_lo, champion_value − margin)
  new_hi = min(orig_hi, champion_value + margin)

Result geometry:
  - Unclamped parameter: search window = 30% of original domain.
  - Clamped at boundary:  window shrinks toward the boundary wall; may be
    < 30% but is always ≥ 0 (guaranteed new_hi > new_lo).
  - 4 log-scale parameters (stop_loss_cents, kelly_fraction,
    max_portfolio_var, drift_vol_ceiling): margin applied in log-space.

For Phase 3 (epsilon lock):
  epsilon = abs(champion_value) × 1e-8
  new_lo  = champion_value − epsilon
  new_hi  = champion_value + epsilon

This forces Optuna's single trial to sample the champion value to within
floating-point precision, making the Phase 3 backtest deterministic.

CHANGE REPORT — Volatility-Adaptive Stop-Loss (Pillar 11.1)
================================================================================

DATE: March 9, 2026

================================================================================
BACKGROUND
================================================================================

The TakeProfit engine (Pillar 3) already dynamically adjusts exit targets based
on rolling volatility, but the StopLossMonitor used a rigid `stop_loss_cents`
parameter.  In high-volatility regimes, background noise repeatedly triggered
the fixed stop before mean-reversion could play out, causing "chop-out" losses.

Pillar 11.1 introduces a volatility-adaptive multiplier that stretches the
baseline stop-loss in high-volatility environments.  Crucially, the multiplier
is clamped so it can **only widen** the stop — it never shrinks below the base

MATHEMATICAL INVARIANTS
================================================================================

The core formula is:

  vol_ratio  = σ_ewma / σ_ref
  multiplier = clamp(vol_ratio, 1.0, max_multiplier)
  SL_trigger = max(1.0, (base_sl × multiplier) − roundtrip_fee_cents)

Four invariants are enforced:

  1. Anti-Shrinkage:  max(1.0, ...) on the left of the clamp ensures that
     when σ_ewma < σ_ref, the multiplier is exactly 1.0.  The stop never
     tightens below the configured baseline.

  2. Max Clamp:  min(..., max_multiplier) on the right of the clamp prevents
     unbounded stretching.  Default max = 1.5 (stop can grow at most 50%).

  3. Missing Data / Cold Start:  When ewma_vol is None, ≤ 0, or is_adaptive
     is False, the multiplier defaults to 1.0.  This preserves legacy
     behavior for markets without enough OHLCV history.

  4. Order of Operations:  The multiplier is applied to the base_sl FIRST,
     THEN fee drag is subtracted.  This ensures the wider window gives
     genuinely more room.  The absolute floor max(1.0, ...) prevents fee
     drag from ever inverting the trigger to a negative or zero value.

CHANGE REPORT — VWAP-Aware Stealth Slicing (SI-4.1)
================================================================================
Date: March 9, 2026

================================================================================
BACKGROUND
================================================================================

The StealthExecutor (SI-4) previously split parent orders into N child slices
using naive time-slicing: a fixed stealth_max_slices cap (default 4) with
randomised delays and ±15% size jitter.  This approach is blind to market

books this leaves a detectable footprint and invites front-running.

The upgrade replaces the fixed slice-count ceiling with a Percentage of Volume
(POV) / VWAP-aware engine that dynamically caps each slice's notional at a
configurable fraction of recent volume, expanding the slice count as needed.

1. Zero-Volume Fallback (Dead Market Invariant):
   If recent_volume_usd ≤ 0 or None, the engine substitutes $10.00 as a safe
   floor.  This prevents division-by-zero and infinite slice expansion.
   Formula: safe_volume = recent_volume_usd if recent_volume_usd > 0 else 10.0

   max_slice_usd = max(safe_volume × stealth_max_participation_pct, min_size_usd)

   required_slices = ⌈notional / max_slice_usd⌉

   Example: $50 notional, $100 volume, 5% cap → max_slice=$5 → 10 slices.
   A structlog event "vwap_stealth_resliced" is emitted on every override.

4. Participation Cap (No-Footprint Invariant):
   In high-volume environments, no individual slice notional exceeds
   recent_volume_usd × stealth_max_participation_pct (within jitter tolerance).
   The existing ±stealth_size_jitter_pct, stealth_abandon_drift_cents, and
   stealth_abandon_fill_pct guards are fully retained.

CHANGE REPORT — Downside Semi-Variance & Adaptive Trailing Stop (Pillar 11.2)
================================================================================

DATE: March 9, 2026

================================================================================
BACKGROUND
================================================================================

Pillar 11.1 introduced a volatility-adaptive stop-loss that stretched the
baseline in high-vol regimes.  Two structural flaws remained:

1. **The "Winning Trade" Flaw**: The vol multiplier used total
   rolling_volatility_ewma.  When the market crashed violently *in our favour*,
   total vol spiked and the stop-loss widened unnecessarily — giving back
   profits that should have been banked.

2. **The "Rigid Trailing" Flaw**: The trailing stop offset was read from
   settings.strategy.trailing_stop_offset_cents (a fixed config value).  In
   high-chop regimes, this rigid offset triggered premature trailing stops,
   cutting winners before mean-reversion completed.

Pillar 11.2 fixes both by:
  - Switching to **downside semi-variance** (only adverse returns contribute).
  - Making the **trailing stop offset** per-position and vol-adaptive.

================================================================================

================================================================================

   - Seeded on first bar with equal-weight downside variance.

2. src/trading/fees.py

     clamp as baseline SL, but NO fee deduction (fees already accounted for
     in the breakeven-activation threshold).
     Returns: base_offset_cents × clamp(σ_down / σ_ref, 1.0, max_mult).

   - `_open_position_inner()`: extracts `rolling_downside_vol_ewma` (not

================================================================================
MATHEMATICAL INVARIANTS
================================================================================

Downside Semi-Variance EWMA:
  downside_r_t = min(r_t, 0.0)     (only negative log-returns contribute)
  σ²_down_t = λ · σ²_down_{t-1} + (1-λ) · downside_r_t²
  rolling_downside_vol_ewma = √(σ²_down_t)

This means:
  - In a pure uptrend (all r_t > 0): downside_r = 0 → σ_down decays to ~0.
  - In a pure downtrend (all r_t < 0): downside_r = r_t → σ_down ≈ σ_total.

Baseline Stop-Loss (unchanged formula, new input):
  multiplier = clamp(σ_down / σ_ref, 1.0, max_multiplier)
  SL_trigger = max(1.0, (base_sl × multiplier) − roundtrip_fee_cents)

Adaptive Trailing Offset (new):
  multiplier = clamp(σ_down / σ_ref, 1.0, max_multiplier)
  trailing_offset = base_offset × multiplier
  (No fee deduction — fees are in the breakeven-activation layer.)

Anti-shrinkage is preserved in both: max(1.0, ...) prevents the multiplier
from going below 1.0, so neither the stop nor the trailing offset can ever
tighten below the configured baseline.

CHANGE REPORT — Burst-Volume & OBI Pacing (SI-4.2)
================================================================================
Date: March 9, 2026

================================================================================
BACKGROUND
================================================================================

SI-4.1 introduced POV/VWAP-aware slice sizing, but two execution flaws remained:

1. Burst-Volume Flaw: scale_probe_to_full() computed recent_volume_usd from

   randomly firing the next slice caused terrible slippage.

SI-4.2 fixes both: Burst-Volume Acceleration uses max(avg, current_bar) volume
so volume spikes accelerate slicing, and Order-Book Imbalance (OBI) Pacing
dynamically biases the sleep between slices based on real-time L2 depth.

   effective_volume = max(avg_bar_volume, current_bar_volume)
   recent_volume_usd = effective_volume × max(scale_price, 0.01)
   During a news spike where current_bar_volume is 20× the average, the
   engine's participation cap widens proportionally, permitting larger
   per-slice notional and fewer total slices.  This prevents over-slicing
   and missed entries in high-opportunity environments.

   book_depth_ratio = bid_depth / ask_depth (from OrderbookTracker)
   For BUY:  our_ratio = book_depth_ratio      (bid is our support)

     our_ratio < 0.5  → hollowed book     → sleep = stealth_delay_max_ms

     0.5 ≤ ratio ≤ 2.0 → balanced         → sleep = random(min_ms, max_ms)

3. Missing Data Fallback:
   If book_depth_ratio is 0, None, or negative → default to 1.0
   If OrderbookTracker is absent or is_reliable=False → default to 1.0
   Both fallbacks produce balanced-zone behaviour (random delays).

CHANGE REPORT — Friction-Responsive Stealth Execution (SI-4.3)
================================================================================
Date: March 9, 2026

================================================================================
BACKGROUND
================================================================================

SI-4.2 introduced Order-Book Imbalance (OBI) Pacing that biases inter-slice
delays based on L2 depth ratios.  However, two reactive execution flaws remained:

1. Slippage Blindness: The stealth loop was open-loop — it never inspected
   whether the *previous* fill actually suffered slippage.  If slice N filled at
   0.51 when we targeted 0.50 (>0.5 ticks friction), the engine happily fired
   slice N+1 at the same tempo.  This compounds execution cost.

2. Iceberg Ignorance: The IcebergDetector already identifies hidden whale
   orders on the opposing side, but the stealth engine did not react.  Firing
   into a detected iceberg is a guaranteed adverse-selection loss because the
   whale will continuously refill, absorbing our flow at stale prices.

SI-4.3 closes both gaps with Slippage-Adaptive Pacing and Iceberg-Aware
Delay Skewing — turning the stealth loop into a friction-responsive system.

================================================================================
FILES MODIFIED
================================================================================

  src/trading/stealth_executor.py — place_stealth_order() accepts
                                    opposing_iceberg_detected: bool param;
                                    loop tracks prev_slippage_choke flag by
                                    comparing filled_avg_price vs target price;
                                    _obi_delay_ms() extended with opposing_iceberg
                                    and slippage_choke kwargs (priority:
                                    slippage_choke > iceberg > OBI ratio)
  src/trading/position_manager.py — scale_probe_to_full() queries
                                    IcebergDetector.strongest_iceberg(opposing_side)
                                    and passes opposing_iceberg_detected=True when
                                    confidence ≥ 0.50
  tests/test_stealth_executor.py  — 21 new tests across 3 classes:
                                    TestSlippageAdaptivePacing (10),
                                    TestIcebergAwareSkewing (7),
                                    TestIcebergWiring (4)

================================================================================
MATHEMATICAL INVARIANTS
================================================================================

1. Slippage-Adaptive Pacing:
   After each FILLED order in the stealth loop:
     slippage_ticks = abs(filled_avg_price - target_price) / 0.01
     prev_slippage_choke = slippage_ticks > 0.5
   If prev_slippage_choke is True → _obi_delay_ms() returns max_delay
   regardless of OBI ratio, giving the book time to refill before next slice.

2. Iceberg-Aware Delay Skewing:
   If opposing_iceberg_detected is True → _obi_delay_ms() returns max_delay
   regardless of OBI ratio.  This throttles execution when a hidden whale
   is absorbing flow on the opposing side.

3. Priority Order in _obi_delay_ms():
     slippage_choke=True  → max_delay  (highest priority)
     opposing_iceberg=True → max_delay  (second priority)
     OBI ratio logic       → normal 3-zone pacing  (lowest priority)

4. Iceberg Detection Wiring:
   PositionManager queries self._iceberg_detectors[asset_id].strongest_iceberg()
   for the opposing side (BUY position → check SELL icebergs).
   Passes opposing_iceberg_detected=True only when confidence ≥ 0.50.
   Graceful fallbacks: missing detector → False, no signal → False,
   confidence < 0.50 → False.

CHANGE REPORT — Covariance-Adjusted Kelly (Pillar 15.1) & Meta-Strategy Controller (SI-6)
================================================================================

BACKGROUND
──────────
Two strategic integration features bring portfolio-level awareness to
position sizing and signal processing:

1.  **Pillar 15.1 — Covariance-Adjusted Kelly**: The Kelly sizer previously
    treated every market as independent.  When the portfolio holds correlated
    positions (ρ > 0.5), the effective variance is higher than the sum of
    individual variances.  We now compute the Marginal Contribution to
    Portfolio Variance (MCPV) — the exposure-weighted average pairwise
    correlation between the proposed trade and all open positions — and scale
    the Kelly fraction by ``(1 − MCPV)``, floored at 0.05:

        f_adj = f* × k_mult × (1 − MCPV)

    This prevents overconcentration in correlated events.

2.  **SI-6 — Meta-Strategy Hybrid Controller**: A regime-aware master switch
    that replaces the binary ``is_mean_revert`` gate with a three-tier
    decision matrix:

      • Deep MR (score ≥ 0.8): 1.5× panic/drift weight, 0.5× RPE.
      • Neutral  (0.3–0.8):    1.0× all signals (passthrough).
      • Trending (score < 0.3): Veto panic/drift ("regime_trend_veto"),
        full RPE pass.

    This refines the existing SI-1 regime gate into graduated weights and
    extends regime awareness to RPE entries.

CHANGE REPORT — SI-5 OFI Wall Pressure Monitor & Preemptive Exit
Date: 2026-03-09
================================================================================

BACKGROUND
----------
OrderChaser (SI-2) pegs orders behind iceberg whale walls to earn maker
priority and reduce market footprint.  However, when a whale's bid wall is
eaten through by a sustained sell burst, our resting order experiences
catastrophic slippage as price falls through the exhausted level with zero
remaining support.

This change extends the existing OFI infrastructure (which only provided a
panic-entry veto) into an active exit trigger: if the opposing order flow
consumed from an iceberg wall exceeds 80% of the whale's estimated remaining
size — or the book's depth ratio drops below 0.15 — the chaser preemptively
cancels the peg and exits 1 tick above the wall before it hits zero.

================================================================================
MECHANISM
================================================================================

Wall Consumption Formula:
  Let E   = iceberg.estimated_total  (refill_count × avg_slice_size)
  Let OFI = L2OrderBook.opposing_ofi_at_price(peg_price, our_side)
            = Σ |negative qty deltas| at that price level in trailing 2 seconds

  Exit trigger fires when:
    OFI / E > ofi_wall_threshold  (default 0.80)       [OFI condition]
    OR book_depth_ratio < ofi_depth_ratio_floor         [depth floor condition]

Exit Execution:
  1. Cancel resting iceberg-pegged limit order.
  2. If partially filled: place market-limit 1 tick from wall
     (best_peg + 0.01 SELL for BUY side; best_peg − 0.01 BUY for SELL side)
     to unwind partial fill at minimal slippage.
  3. Set chaser state → ABANDONED.

================================================================================

================================================================================
DESIGN DECISIONS
================================================================================

Preemptive exit scope: iceberg-pegged chasers only.
  Only chasers with _iceberg_peg_active=True are monitored. Plain BBO-quoted
  chasers have no whale wall to lose and no whale-size estimate to compare
  against. Adding preemptive exit to plain chasers would require a different
  triggering heuristic.

  book.opposing_ofi_at_price is checked via getattr() — missing on
  OrderbookTracker, present on L2OrderBook. When absent, _should_preemptive_exit
  skips the OFI condition and only the depth_ratio floor path remains active.

1-tick exit price (not market order).
  _preemptive_exit() places a limit order 1 tick from the wall with
  post_only=False (taker). On Polymarket's crowded CLOB, a 1-tick-inside
  limit will in practice fill immediately at taker cost, but avoids the
  risk of an unlimited-slippage true market order.

Depth ratio floor (condition 2) check first.
  The depth floor check is O(1) (single float comparison) and fires
  correctly even when no iceberg detector is wired. It therefore acts as a
  fallback safety net against sudden book thinning unrelated to iceberg exhaustion.

CHANGE REPORT — Stink-Bid Cascade Harvesting (Liquidity Gap Exploitation)
Date: 2026-03-09
================================================================================

BACKGROUND
----------
Retail traders and basic bots cluster stop-losses at "liquidity gap" levels —
price points where the ask-side order book is unusually thin.  When a panic
event drives selling, these stops cascade through the gaps, creating brief
price dislocations below fair value.

This change introduces two new capabilities to exploit this dynamic:
  1. L2OrderBook.find_liquidity_gaps() — identifies thin ask levels that
     could be wiped by a stop-loss cascade.
  2. PositionManager.harvest_cascades() — places passive "stink bid" limit
     orders at those gaps when a PanicSignal fires, targeting fills that only
     occur when the cascade actually sweeps through.

Stink bids are passive entries at better-than-standard prices — they do NOT
chase the current best ask.  They sit 2–4 ticks deeper and are filled only
when retail stop-losses flush inventory through thin book levels.

================================================================================
FILES MODIFIED
================================================================================

1. src/data/l2_book.py
   Added method: L2OrderBook.find_liquidity_gaps(min_depth_usd: float = 0.0)

   Algorithm:
   - Computes 5-minute average total depth from _depth_history (300-second
     window cutoff).  avg_depth = mean of all samples in the window.
   - threshold = max(min_depth_usd, avg_depth × 0.10)
     → If threshold ≤ 0 (no depth history), returns [] immediately.
   - Iterates ask side via self._asks.islice(stop=self._max_levels),
     accumulating cumulative USD depth (price × size) level by level.
   - Appends a Level(price, cumulative_depth_usd) for each level where
     the running cumulative is below the threshold.
   - Returns list[Level] in ascending price order.

   Design notes:
   - Uses the existing _depth_history deque (populated by _record_depth
     on every BBO change) — zero new storage overhead.
   - The 10%-of-average threshold is self-calibrating: wider books require
     deeper gaps to qualify, thinner markets are more sensitive.
   - Returns [] on cold-start (no depth history), preventing false signals
     immediately after snapshot load.

2. src/trading/position_manager.py
   Added method: PositionManager.harvest_cascades(
       signal: PanicSignal,
       l2_book: Any,
       *,
       event_id: str = "",
       min_depth_usd: float = 0.0,
       tick_offset_range: tuple[int, int] = (2, 4),
   ) -> list[Position]

   Algorithm:
   - Reads best_ask reference from signal.no_best_ask (the live market level
     at signal-fire time — NOT l2_book.best_ask, which may already have moved).
   - Calls l2_book.find_liquidity_gaps(min_depth_usd) to get thin levels.
   - Computes tick window:
       lower_bound = round(best_ask − hi × 0.01, 2)
       upper_bound = round(best_ask − lo × 0.01, 2)
     Default (2,4): 2–4 ticks below best ask.
   - Filters gaps to those within [lower_bound, upper_bound] with price > 0.
   - Acquires _entry_lock so all stink bids are placed atomically with
     standard position entries (prevents race conditions).
   - For each qualifying gap level:
     a. Calls _check_risk_gates(signal.market_id, event_id) — aborts
        loop on circuit breaker trip or hard gate failure.
     b. entry_size = max(1.0, round((max_trade × 0.25) / entry_price, 2))
        → 25% of capital allocation per level (conservative passive sizing).
     c. Calls executor.place_limit_order() at gap price.
     d. Constructs Position with id prefix "STINK-", state=ENTRY_PENDING.
     e. Logs "stink_bid_placed" structlog event.
   - Returns list of created Positions (empty list if no gaps/gate fails).

   Intentional design decisions:
   - Uses signal.no_best_ask (not book.best_ask) as the window anchor.
     Rationale: the book may have partially moved by the time harvest_cascades
     is called; the original signal level is the correct cascade reference.
   - No EQS scoring on stink bids. Rationale: gap levels are below the
     standard ask-minus-1¢ entry — EQS would be circular.  Risk gates
     (VaR, concentration, daily loss) still apply.
   - Sizing is 25% of max_trade per level (not Kelly). Rationale: stink bids
     have low fill probability; over-sizing passive orders wastes the VaR budget.

CHANGE REPORT — Technical Debt Remediation: Seqlock IPC, Synthetic Market
Impact, Circuit Breaker Resilience
Date: March 9, 2026
================================================================================

OVERVIEW
--------
Three "band-aids" introduced during the March 5th–9th deployments were
identified and fully remediated:

  Band-aid 1 — IPC seqlock had a spin-lock remnant and raised IPCReadError on
               contention, which is incompatible with true lock-free design.

  Band-aid 2 — The backtest matching engine consumed unlimited historical
               liquidity on every tick, letting Optuna discover overfit
               parameters that exploit infinite BBO depth.

  Band-aid 3 — The trade-loop circuit breaker responded to WebSocket-induced
               errors with a fatal bot.stop(), unnecessarily killing the entire
               process on transient network instability.

================================================================================
CHANGE 1 — TRUE LOCK-FREE SEQLOCK IPC
================================================================================

File: src/core/ipc.py

PROBLEM:
  A previous patch retained a write_lock spin-lock fallback (100-iteration cap)
  alongside the Seqlock, and introduced IPCReadError to short-circuit on first
  reads that exhausted retries.  These were contradictions of a lock-free
  design: a spin-wait in the writer is blocking, and raising an exception
  forces callers to handle a contention case that should never occur with true
  double buffering.

CHANGES:
  a) Removed IPCReadError exception class entirely from ipc.py.
     - Only external reference was a comment in scripts/stress_test_seqlock.py
       (also updated there).
     - No production import site was using it.

  b) Removed _contention_warnings integer counter from SharedBookReader.__slots__.
     - This was the analog of the old _spin_failures counter.

  c) Simplified the "all retries exhausted" code path in both read_header()
     and read_full():
     - OLD: increment _contention_warnings; log high_contention_warning every
       100 hits; if _last_snapshot exists return it, else raise IPCReadError.
     - NEW: if _last_snapshot exists return it, else return
       SharedBookSnapshot(asset_id=self.asset_id) — an empty but valid
       snapshot with all defaults.
     - This path is near-impossible in normal operation (double-buffering
       guarantees a stable block is always readable), but is now fully
       non-throwing for the rare contention case.

  d) Updated scripts/stress_test_seqlock.py crash-recovery test:
     - Removed comment referencing IPCReadError as expected exception.
     - Updated got_data check: snap.seq > 0 (non-empty) instead of
       relying on exception semantics.

DESIGN NOTE:
  The Seqlock protocol itself (writer increments seq odd → writes standby
  block → flips active index → increments seq even; reader checks s1 even →
  reads active block → re-checks s2 == s1) was already correct and required
  no changes.  Only the error handling layer was cleaned up.

CHANGE 2 — SYNTHETIC MARKET IMPACT MODEL (BACKTESTER)
================================================================================

File: src/backtest/matching_engine.py
File: src/backtest/engine.py

PROBLEM:
  The WFO backtester fed historical L2 snapshots into the matching engine
  and then allowed simulated orders to consume liquidity from those snapshots
  at face value on every tick.  When the next historical snapshot arrived, the
  book was fully restored from historical data regardless of how much the
  simulated strategy had already consumed.  This gave the backtester access to
  effectively infinite liquidity at the BBO, causing Optuna to over-select
  parameters suited for aggressive, high-fill-rate strategies that would
  suffer severe real-world market impact.

CHANGES:
  a) New dataclass _LiquidityDebt (slots=True):
       volume: float    — shares consumed in a past simulated fill
       timestamp: float — sim-time the fill occurred

  b) New MatchingEngine constructor parameter:
       impact_recovery_ms: float = 5000.0
     Stored as self._impact_tau = impact_recovery_ms / 1000.0 (decay τ in
     seconds for exponential recover curve).

  c) New instance attributes:
       self._ask_debt: dict[float, list[_LiquidityDebt]]  — per ask-price debts
       self._bid_debt: dict[float, list[_LiquidityDebt]]  — per bid-price debts
       self._last_time: float = 0.0                       — for τ seeding

  d) on_book_update() signature extended with current_time: float | None = None.
     After applying the historical delta/snapshot, _apply_liquidity_debt() is
     called at the provided timestamp before _refresh_bbo().

  e) _apply_liquidity_debt(current_time):
     Iterates all entries in _ask_debt and _bid_debt.  For each price level:
       1. Computes the total decayed debt:
            debt = Σ d.volume × exp(−(now − d.timestamp) / τ)
       2. Subtracts debt from the raw historical book level:
            adjusted = raw_size − debt
          If adjusted ≤ 0, removes the level entirely from the book.
       3. Prunes _LiquidityDebt entries whose remaining contribution falls
          below 1% of original volume (exp decay to < 0.01).
     This means the book levels seen by subsequent order activations already
     reflect the simulated strategy's market footprint.

  f) _execute_taker() updated (both BUY and SELL paths):
     Before computing fill_qty:
       avail = max(0.0, raw − self._decayed_debt(debts_at_price, now))
       if avail < 1e-9: continue  ← skip exhausted level
     After computing fill_qty:
       self._record_debt(debt_map, price, fill_qty, current_time)
     Debt accumulates across multiple orders at the same level.
     self._last_time updated on every _execute_taker call.

  g) reset() now also clears _ask_debt and _bid_debt.

  h) src/backtest/engine.py:
     Both on_book_update() call sites (real L2 events and synthetic BBO
     construction) updated to pass current_time=event.timestamp.

DECAY FORMULA:
  Available size at price p at time t =
    historical_size(p) − Σ_i [v_i × exp(−(t − t_i) / τ)]

  where v_i is the fill volume and t_i is the fill timestamp.
  With τ = 5.0 s: 50% recovery at 3.5 s, 99% recovery at 23 s.
